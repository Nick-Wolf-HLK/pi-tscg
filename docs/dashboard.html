Architektur

Drei Hebel, drei Hooks

Pi exponiert die volle Agent-Lifecycle als Event-Hooks. Diese Extension hängt sich an drei davon — zwei provider-agnostisch, einer provider-aware.

Hebel 1

Tool-Schema-Compression

Hängt sich an before_provider_request und schickt jedes Tool-Schema durch compressDescriptions() mit allen acht TSCG-Operatoren (SDM, DRO, CAS, CCP, CFL, CFO, SAD, TAS). Tool-Namen bleiben byte-identisch, Dispatch ist unverändert.

Plus: Strategy A — JSON-Pruning entfernt non-semantic Validator-Metadaten. $ref und additionalProperties bewusst behalten (semantisch).

8 - 50%
je nach Tool-Catalog (Pi-Built-ins → MCP-Tools)
Hebel 2

Tool-Result-Compression

Hängt sich an tool_result und kürzt Tool-Output, bevor er ans Modell zurück geht. Per-Tool-Pipeline: Whitespace-Collapse → Duplicate-Line-Folding → SDM-Filler-Removal → head/tail-Truncation. Edits bleiben byte-exakt unangetastet.

Token-Budget per Default 4 000. Truncation explizit markiert: [TSCG: N lines / ~Xk tokens omitted — head + tail preserved]

40 - 70%
dominant bei großen bash/find/read Outputs
Hebel 3

Provider-Cache-Awareness

Erkennt den Provider aus dem Tool-Format und engaged dessen Cache-Mechanik. Anthropic: setzt cache_control: ephemeral. OpenAI: implizit aktiv, weil unsere deterministische Compression das Auto-Prefix-Caching triggert. Ollama: no-op. Gemini: Skeleton-Hook PR-ready.

~10x
Anthropic-Tools werden ab Call 2 zu ~10% des Preises abgerechnet
Hebel 3 · Provider-Awareness

Pro Provider eine eigene Cache-Mechanik

Cache-Mechanismen sind technisch grundverschieden. Hebel 3 ist deshalb ein Switch über den erkannten Provider — kein One-Size-Fits-All.

Anthropic
Claude API
Setzt cache_control: ephemeral auf das letzte Tool im Block. Anthropic cached den ganzen Tools-Array. Ab Call 2: ~10% Kosten.
Live ✓
OpenAI
GPT API · ≥1024 Tokens
OpenAI cached automatisch identische Prefixes. Wir setzen keinen Marker, profitieren aber von TSCG's Determinismus → Cache-Hit ab Call 2.
Implizit ✓
Ollama
localhost:11434
Lokales Modell, kein Token-Pricing, kein API-Cache. Korrekter no-op — Hebel 1 + 2 reichen, weil Compression hier Inferenzzeit spart, nicht Kosten.
No-op

Was die Praxis-Integration für TSCG hergibt

Drei konkrete Anregungen aus dem Pi-Lauf, die anderen Adapter-Autoren auch helfen würden.

compressFreeText API

Tool-Output-Compression als first-class Use-Case neben Tool-Schema-Compression. Würde Hebel 2 in jeder Pi/Aider/Continue-Integration trivial machen.

mapPiModelId als Adapter

Die Mapping-Map von Pi-Modell-IDs zu deinen ModelTarget-Werten könnte ich als PR ins examples/ einbringen — Referenz für andere Adapter-Autoren.

outputFormat Doku

Was outputFormat: "compact" im description-only Mode konkret tut, ist nicht klar. Ein Beispiel-Snippet im README würde Adopters helfen.

Distribution

Drop-in via npm

Sobald veröffentlicht: ein Befehl reicht, um alle drei Hebel live in jeder Pi-Session zu aktivieren — keine Code-Änderung am Agent, kein Cluster-Setup, kein Account.

$ pi install pi-tscg
→ resolves @tscg/core, registers extension, ready in footer
